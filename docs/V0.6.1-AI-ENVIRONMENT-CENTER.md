# V0.6.1 本机 AI 环境中心

## 目标

环境中心只解决三件事：知道本机哪些 AI 可以用、让使用者选择是否处理缺失状态、处理后验证结果是否真实生效。它不替代 CLI，不保存账号凭证，也不把安装过程放进聊天区。

## 产品行为

- AI 工作台加载时检测 Codex、Claude Code、Kimi Code、Antigravity 与 Grok Build。
- 已安装、已登录且具备当前连接能力的 Provider 直接出现在新会话下拉框。
- “管理本机 AI”打开紧凑抽屉，显示本机版本、驾驶舱已验证版本、状态和官方说明。
- 未安装显示“安装”，登录缺失显示“登录”，本机版本低于登记的上游版本显示“更新”。
- 安装、更新和登录都先由使用者点击，再经过浏览器确认。
- 安装或更新结束后，服务端重新执行完整探测。CLI 仍然缺失，或更新前后版本没有变化时，任务失败并显示真实原因。

## Provider 合同

| Provider | 新会话入口 | 连接方式 | 环境动作 |
|---|---|---|---|
| Codex | `codex` | 固定 ACP 适配器 | 官方 npm 包；本机登录 |
| Claude Code | `claude` | 固定 ACP 适配器 | 官方 npm 包；本机登录 |
| Kimi Code | `kimi` | 原生 ACP | 官方 npm 包；本机登录 |
| Antigravity | `agy` | 官方打印与会话恢复接口 | 官方安装程序；官方 `update`；本机登录 |
| Grok Build | `grok` | 原生 stdio | 官方 npm 包；本机登录 |

旧 `gemini` 仍在历史 Conversation 和 Run Schema 中可读，但不出现在探测列表，也不能创建新会话。

## Antigravity 会话

Antigravity 没有复用 ACP Runner。每轮使用固定参数进入 `plan` 模式并启用 `sandbox`，首轮从受限日志中提取官方 Conversation ID，后续轮次使用同一 ID 恢复上下文。

- 浏览器消息作为单个 `spawn` 参数传入，`shell:false`，不经过 shell 插值。
- 首版只允许 `readonly`，不接受网页写入授权。
- 标准输出上限 2 MiB，诊断上限 16 KiB，会话日志上限 2 MiB。
- 输出、错误与事件在公开前脱敏；临时日志在每轮结束后删除。
- 取消或关闭会话会回收本产品启动的进程组。
- Provider Session ID 只保存在服务端会话状态，不进入公开 DTO。

## 环境操作边界

API：

- `POST /api/ai-environment/actions`
- `GET /api/ai-environment/actions/:jobId`

POST 请求只接受：

```json
{
  "provider": "codex | claude | kimi | antigravity | grok",
  "action": "install | update | login"
}
```

浏览器不能提交可执行路径、shell 命令、包名、安装地址、参数或环境变量。所有动作来自服务端固定表，并满足以下边界：

- 仅接受本机回环、同源、带 CSRF 标记的请求。
- macOS 首发；其他平台拒绝环境操作，但不影响只读页面。
- 同一时间最多一个环境任务。
- 未安装 Provider 不能更新或登录；已安装 Provider 不能重复安装。
- 当前版本不能重复更新；已登录 Provider 不能重复打开登录流程。
- Provider 拥有未关闭会话或正在执行 Turn 时，不允许安装、更新或登录。
- 登录只打开固定临时 Terminal 流程；驾驶舱不读取登录输入或凭证。
- 任务公开状态不包含 stdout、stderr、脚本正文、包名、路径或 Token。

## 验收口径

V0.6.1 的发布检查包括：

- 五个 Provider 的本机安装、版本、能力与登录探测。
- 环境 API 的回环、同源、CSRF、严格字段和固定命令测试。
- 未生效安装、无变化更新、重复操作和活动会话冲突测试。
- Antigravity 两轮真实会话测试：第二轮必须能引用第一轮随机标记，工作区不能产生未授权文件。
- 前端管理抽屉、动作确认、Provider 下拉框、1024/1280/1440 桌面布局和浏览器控制台检查。

V0.6.1 不承诺多用户、Windows/Linux 安装、无人值守升级、凭证托管、自动修复登录或多 Agent 编排。
