# V0.4 本机 AI 协作工作台

## 产品目标

V0.4 只解决一个问题：让创作者从驾驶舱里选择一项真实内容任务，调用已经安装并登录的本机 AI CLI，在清楚的权限边界内完成工作，并把人工确认后的结果保存回 Obsidian。

这一版本不是网页终端，也不是无人值守的多 Agent 系统。使用者始终能够看到 Agent 正在做什么、拒绝高风险动作、取消运行，并决定结果是否进入第二大脑。

## 首版闭环

```text
选择现有内容上下文
        ↓
选择任务模板和本机 Agent
        ↓
复制上下文到独立临时任务空间
        ↓
通过 ACP 启动单个 Agent
        ↓
流式显示回复、计划、工具调用与状态
        ↓
敏感动作逐次批准或拒绝
        ↓
人工确认最终结果
        ↓
受控写入 Obsidian 项目工作区
```

## 架构参考与取舍

- 进程通信采用 [Agent Client Protocol](https://github.com/agentclientprotocol/agent-client-protocol)（Agent 客户端协议，ACP）稳定协议，而不是为每个 CLI 自定义一套聊天接口；运行时记录双方协商出的 `protocolVersion`。
- Codex 通过 [codex-acp](https://github.com/zed-industries/codex-acp) 适配，Claude Code 通过固定本机 ACP 适配器接入；两者均不允许页面临时下载或执行未知适配器。
- Kimi 使用官方文档定义的 [`kimi acp`](https://moonshotai.github.io/kimi-code/en/reference/kimi-acp.html) 入口。
- Gemini 以官方 [Gemini CLI](https://github.com/google-gemini/gemini-cli) 为能力边界；认证不可用时明确降级，不伪装成可运行。

这些项目只提供协议和适配层参考。驾驶舱自身仍负责上下文白名单、临时工作区、权限确认、事件留痕、人工交付和 Obsidian 受控写回，不能把 CLI 的可调用等同于产品已经安全可用。

## 支持的 Agent

| Provider ID | 页面名称 | ACP 入口 | 本机 CLI |
|---|---|---|---|
| `codex` | Codex | `codex-acp` 适配器 | `codex` |
| `claude` | Claude Code | `claude-agent-acp` 适配器 | `claude` |
| `kimi` | Kimi Code | `kimi acp` | `kimi` |
| `gemini` | Gemini CLI | `gemini --acp` | `gemini` |
| `grok` | Grok Build | `grok agent stdio` | `grok` |

服务端只允许这五个固定 Provider。页面不能提交可执行文件、启动参数、环境变量或任意 shell 命令。

## 任务模板

首批提供八个模板：

1. 分析一个选题。
2. 拆解一条内容。
3. 生成文章草稿提案。
4. 生成短视频草稿提案。
5. 复盘一条内容。
6. 拆解一个对标账号。
7. 汇总当天整体复盘。
8. 根据复盘提出明日行动。

模板只负责预填任务目标。使用者仍可补充本次具体要求，不能由模板隐式扩大权限。

```yaml
id:

status: queued | running | waiting_permission | completed | failed | cancelled

  type:
  id:

  summary:

  providerVersion:
  adapterPackage:
  adapterVersion:
  protocolVersion:
  versionStatus: current | outdated | newer | unknown

```

这是浏览器从本机 API 收到的公开对象。工作目录、可执行文件、原始来源路径和 Provider 会话 ID 不向浏览器暴露；它们只在需要时保留为服务端运行证据。运行元数据进入本地 SQLite；完整事件同时写入当前运行目录的有界 JSONL。原始会话记录默认不进入 Obsidian。

## 本地目录

```text
{COCKPIT_STATE_ROOT}/ai-runs/
└── {run_id}/
    ├── inputs/
    │   └── context.md
    ├── outputs/
    ├── manifest.json
    └── events.jsonl
```

任务目录必须位于固定状态根。创建前后都检查真实路径，拒绝绝对路径注入、`..`、软链接、非目录节点和越界文件。

## 权限模型

- 浏览器只提交资料类型与资产 ID；服务端从权威索引解析标题、摘要、原文路径和 SHA-256，并在复制前复验常见云端密钥与私钥，再把经过白名单校验的真实 Markdown 副本放入临时任务空间。浏览器不能伪造来源正文或证据链。
- 默认 `readonly`。Codex 强制使用原生 `read-only` 会话模式，Claude Code 强制使用原生 `plan` 会话模式；Gemini 在认证恢复后使用官方 `plan` 启动模式。
- Kimi Code 与 Grok Build 当前 ACP 入口没有可强制的原生只读会话模式。驾驶舱仅开放“只读分析”，拒绝它们发回的写入权限请求，不提供网页授权写入；提示词仍只是协作约束，不是操作系统级隔离。
- `ask` 模式下，ACP 上报的写文件、删除、运行命令和联网请求必须逐次确认；没有上报给 ACP 的进程行为无法由网页授权框拦截。
- 页面只显示 ACP 返回的真实权限选项；`allow_always` 和 `reject_always` 不提供给首版用户。
- 允许只对当前工具调用有效，超时按拒绝处理。
- 驾驶舱不会把 `.git`、CLI 配置、身份规则、shell profile 或 Vault 系统规则作为任务工作区，也不会为这些位置提供网页授权；当前没有操作系统级沙箱，因此不能保证失控进程无法主动访问这些本机文件。
- 同时最多运行两个 Agent；同一 run 严格串行。
- 取消顺序为 ACP cancel、短暂等待、终止当前进程组，最后才强制回收该进程组。

当前子进程仍以本机登录用户身份运行并继承必要的 `HOME` 等配置，以复用官方 CLI 登录状态。独立任务目录不是系统沙箱，不能阻止恶意或失控进程主动访问本机其他文件。因此首版适合本人在本机、可见操作下使用，不应开放公网，也不应把未知脚本或不可信内容交给 Agent 执行。

## 进程边界

- 只使用服务端 Agent Catalog 中解析并复验过的绝对路径。
- 使用 `spawn(executable, argv, { shell: false })`。
- Prompt 通过 ACP 消息发送，绝不拼接进 shell 参数。
- 子进程只继承最小环境变量白名单。
- Codex 适配器通过 `CODEX_PATH` 指向已验证的本机 Codex。
- Claude 适配器通过 `CLAUDE_CODE_EXECUTABLE` 指向已验证的本机 Claude Code。
- 登录仍由用户在官方 CLI 中完成。状态接口显示安装、版本和 ACP 启动能力；登录及握手状态以真实运行结果为准。

## Obsidian 写回

驾驶舱不会把 Vault 作为任务工作区，也不会在使用者点击“确认保存”之前触发受控导入：

```text
{projectRelativeDir}/03-工作过程/AI协作/
```

写回文件包含来源 run、Provider、版本、任务上下文、确认时间和最终文本。服务端使用固定目录、原子替换、外部备份、SHA-256 冲突检查和写后索引校验。失败时恢复旧文件；若进程在文件落盘后中断，幂等恢复会重新刷新索引，失败时保留已验证文件并允许安全重试。

## HTTP 接口

- `GET /api/ai-agents`：五个 Agent 的安装、版本和 ACP 启动能力；登录及握手状态以真实运行结果为准。
- `GET /api/ai-runs`：最近运行。
- `POST /api/ai-runs`：创建并启动运行。
- `GET /api/ai-runs/:runId`：运行快照。
- `GET /api/ai-runs/:runId/events`：SSE 事件流。
- `POST /api/ai-runs/:runId/cancel`：取消运行。
- `POST /api/ai-runs/:runId/permissions/:permissionId`：批准或拒绝一次权限请求。
- `POST /api/ai-runs/:runId/import`：人工确认后写入 Obsidian。

所有接口只接受回环地址与同源请求。写接口必须通过 CSRF、JSON 类型、大小、字段和状态机校验。

## 首版明确不做

- 任意网页终端。
- 自动安装、升级、登录或读取凭证。
- 多 Agent 自动分工和 Agent 互相调用。
- 五个 Agent 同时改同一份文件。
- 无人值守目标、定时任务和自动发布。
- 公网、局域网或手机远程控制。
- 永久授权和一键 YOLO。
- 直接向整个 Vault 开放读写权限。

## 发布验收

1. 五个 Provider 均完成命令与版本检测；Codex、Claude Code、Kimi Code、Grok Build 完成真实 ACP 最小运行，Gemini 明确呈现当前认证失败原因。
2. 缺失、版本落后、未登录、超时和异常退出都能稳定显示。
3. Mock ACP 覆盖正常流、权限拒绝、单次批准、取消、崩溃、无效 JSON 和输出超限。
4. Prompt 中的 shell 特殊字符只作为文本传递。
5. 路径穿越与软链接逃逸均被拒绝，外部目录无变化。
6. 页面刷新后可恢复运行快照与事件序号。
7. 人工确认前 Vault 无变化；确认后原子写入并重建索引。
8. 可登录的本机 Agent 各完成一次最小只读真实运行；未登录或认证方式被停用的 Agent 明确显示阻塞原因，不尝试代替用户登录。
9. 全量前端、服务端、索引、类型、构建和依赖安全测试通过。
